# Автопубликация @allindsgn — Telegram + Threads

Публикует посты №18–35 из контент-плана строго по датам из календаря
(пн/ср/пт, 21 июля — 29 августа 2026), но **ничего не уходит без твоего
подтверждения**. Схема:

1. За ~1 час до времени поста бот присылает тебе в личку превью текста.
2. Публикация (job `publish`) **встаёт на паузу** и ждёт, пока ты нажмёшь
   **Approve** в GitHub — это делается прямо с телефона (push от GitHub
   мобильного приложения) или по ссылке из уведомления в Telegram.
3. Только после Approve пост уходит в Telegram-канал и в Threads.

Если ты ничего не одобришь — пост просто останется висеть в очереди,
ничего не опубликуется само.

---

## Что уже готово в этом архиве

```
.github/workflows/publish.yml   — расписание + логика уведомление→approve→публикация
data/posts.json                 — посты 18–35, разобранные из твоего контент-плана
data/photo_catalog.json         — коды фото → имена файлов
scripts/notify.py               — шлёт превью в личку
scripts/publish.py              — публикует в Telegram-канал и Threads
scripts/tz_utils.py             — время Ташкента (UTC+5)
assets/photos/                  — СЮДА нужно положить сами фото (см. ниже)
```

---

## Шаг 1. Создать репозиторий на GitHub

1. Создай новый репозиторий, например `allindsgn-autopost`.
   Можно **приватный** — секреты и approve работают одинаково.
   Публичным репозиторий нужен только если хочешь публиковать фото в
   Threads (см. Шаг 4) через `raw.githubusercontent.com`.
2. Залей туда содержимое этого архива целиком, сохранив структуру папок.
3. Добавь сами фото в `assets/photos/` с именами как в
   `data/photo_catalog.json` (`photo_2026-06-09_11-43-24.jpg` и т.д.) —
   их у меня нет, только имена файлов из твоего плана.

---

## Шаг 2. Создать Telegram-бота и получить свой chat_id

1. В Telegram открой **@BotFather** → `/newbot` → задай имя и username
   (например `allindsgn_publisher_bot`). Получишь **токен** вида
   `123456:AA...` — это `TELEGRAM_BOT_TOKEN`.
2. Добавь этого бота **администратором** в канал `@allindsgn`
   (Канал → Управление → Администраторы → Добавить).
3. Узнай свой личный `chat_id` (куда бот будет слать превью тебе):
   - Напиши боту **@userinfobot** в личку — он пришлёт твой numeric ID.
   - Или напиши что угодно своему новому боту, затем открой
     `https://api.telegram.org/bot<ТВОЙ_ТОКЕН>/getUpdates` в браузере —
     там будет `"chat":{"id": ...}`.
   Это и есть `TELEGRAM_OWNER_CHAT_ID`.
4. `TELEGRAM_CHANNEL_ID` — это `@allindsgn`.

## Шаг 3. Добавить секреты в GitHub

В репозитории: **Settings → Secrets and variables → Actions → New repository secret**:

| Имя | Значение |
|---|---|
| `TELEGRAM_BOT_TOKEN` | токен из BotFather |
| `TELEGRAM_OWNER_CHAT_ID` | твой личный chat_id |
| `TELEGRAM_CHANNEL_ID` | `@allindsgn` |
| `THREADS_ACCESS_TOKEN` | см. Шаг 4 (можно добавить позже) |
| `THREADS_USER_ID` | см. Шаг 4 (можно добавить позже) |

Без двух последних скрипт просто пропустит публикацию в Threads и
опубликует только в Telegram — ничего не сломается.

## Шаг 4. Настроить Threads API (можно сделать позже, отдельно)

Это самая муторная часть, делается один раз:

1. Убедись, что твой аккаунт Threads переведён в **профессиональный**
   (Business/Creator) — в приложении Threads: Настройки → Аккаунт.
2. Зайди на **developers.facebook.com** → создай приложение → тип
   **"Other" → "Business"** → добавь продукт **Threads API**.
3. В настройках приложения добавь себя как **Threads tester**
   (App roles → Roles → Threads testers) и прими приглашение в
   уведомлениях Threads-приложения на телефоне.
4. Через **Graph API Explorer** или OAuth-flow приложения получи
   **short-lived access token** с правами
   `threads_basic,threads_content_publish`.
5. Обменяй его на **long-lived token** (действует 60 дней) запросом:
   ```
   GET https://graph.threads.net/access_token
     ?grant_type=th_exchange_token
     &client_secret=<APP_SECRET>
     &access_token=<SHORT_LIVED_TOKEN>
   ```
6. Узнай свой `THREADS_USER_ID` запросом:
   ```
   GET https://graph.threads.net/v1.0/me?fields=id,username&access_token=<TOKEN>
   ```
7. Обе величины положи в секреты `THREADS_ACCESS_TOKEN` и `THREADS_USER_ID`.

⚠️ Long-lived токен живёт 60 дней и его нужно обновлять (запрос
`grant_type=th_refresh_token`) — иначе публикация в Threads молча
перестанет работать примерно через 2 месяца. Если хочешь, я допишу
отдельный workflow, который сам обновляет токен раз в 50 дней и
переписывает секрет через GitHub API — просто скажи.

**Фото в Threads** публикуются только по публичной ссылке (не файлом).
Если репозиторий публичный, добавь переменную (не секрет, а
**Variable**) `PUBLIC_ASSETS_BASE_URL` =
`https://raw.githubusercontent.com/ТВОЙ_ЮЗЕР/ТВОЙ_РЕПО/main/assets/photos`.
Без неё Threads-посты будут уходить как текст без картинки.

## Шаг 5. Настроить Environment с обязательным подтверждением

Это и есть тот самый "approve перед публикацией":

1. **Settings → Environments → New environment** → назови `production`
   (имя должно совпадать с `environment: production` в workflow).
2. Включи **Required reviewers** → добавь себя (свой GitHub-аккаунт).
3. Сохрани.

Теперь job `publish` в workflow будет **виснуть и ждать** твоего
нажатия Approve в интерфейсе Actions (или в мобильном приложении
GitHub, если оно у тебя установлено и включены уведомления).

---

## Шаг 6. Проверка

- Вручную запусти workflow: **Actions → Публикация постов allindsgn
  (пн/ср/пт) → Run workflow**. Это не будет привязано к реальной дате
  поста, но проверит, что бот и секреты работают (если сегодня не
  пн/ср/пт с постом в очереди — просто ничего не найдёт и завершится
  без действий, это нормально).
- В боевом режиме ничего делать не нужно — расписание сработает само
  по датам из `data/posts.json`.

---

## Как это ведёт себя по факту

- Даты постов **не сдвигаются** — они зафиксированы в `data/posts.json`
  ровно как в твоём календаре (21 июля, 23 июля, 25 июля … 29 августа).
- Время публикации = момент, когда ты нажал Approve (обычно в течение
  часа от исходного времени 09:00/11:00/15:00, если реагируешь
  оперативно на уведомление).
- Если пост не одобрен — он остаётся в очереди со статусом `notified`
  и НЕ публикуется автоматически позже. Хочешь другую логику (например,
  автоотмену через N часов) — скажи, добавлю.

## Что нужно доделать тебе

1. Закинуть архив в новый GitHub-репозиторий.
2. Положить фото в `assets/photos/`.
3. Пройти Шаги 2–3 (бот + секреты) — без этого не заработает даже
   Telegram-часть.
4. Threads (Шаг 4) — можно отложить, всё остальное при этом работает.
5. Шаг 5 (environment `production` + required reviewer) — без этого
   approve-паузы не будет, и посты будут публиковаться сразу без
   подтверждения.
